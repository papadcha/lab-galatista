## v1.1.27 — 2026-07-05

**Ενισχύσεις ακεραιότητας δεδομένων + διορθώσεις από code review (Gemini + ChatGPT)**

- **Νέο: `PRAGMA integrity_check` μετά από κάθε backup** — αν το φρέσκο backup είναι κατεστραμμένο, απορρίπτεται αμέσως αντί να αποθηκευτεί σιωπηλά ως έγκυρο.
- **Fix: επιβεβαίωση στο Clean Start όταν αποτυγχάνει το cloud sync** — αν το FINAL backup δεν ανέβει στο cloud, εμφανίζεται προειδοποιητικός διάλογος πριν προχωρήσει η διαγραφή δειγμάτων.
- **Fix: Archive Mode "split-brain" μετά από crash** — αν η εφαρμογή κλείσει απότομα ενώ είναι σε Archive Mode, το `archiveDataFolder` έμενε μόνιμα στο config, οδηγώντας backups/PDF σε λάθος (αρχειοθετημένο) φάκελο ενώ η βάση θα ήταν η ζωντανή. Προστέθηκε καθαρισμός στην εκκίνηση.
- **Fix: σιωπηλό masking σφάλματος στην ημερομηνία λήξης CE** — αν η ημερομηνία λήξης ήταν κατεστραμμένη/μη έγκυρη, το σύστημα επέστρεφε "όλα καλά" αντί να προειδοποιήσει. Τώρα εμφανίζεται ορατή προειδοποίηση.
- **Fix: SQL injection pattern στο `vacuum_into`** — το path του backup περνούσε σε SQL string χωρίς escaping· θα έσπαγε αν ο φάκελος δεδομένων περιείχε απόστροφο.
- **Fix: λείπον semantic έλεγχος ημερομηνιών στην Περιοδική Αναφορά** — δέχονταν μορφότυπα σαν "32/13/2026" χωρίς έλεγχο εγκυρότητας ημερολογίου.
- **Fix: προστασία διπλού-κλικ στην παραγωγή PDF** (Αρχική + Εκθέσεις) — δύο σχεδόν ταυτόχρονες κλήσεις μοιράζονται πλέον την ίδια σε-εξέλιξη παραγωγή αντί να ξεκινούν περιττή δεύτερη.
- Cleanup: αφαιρέθηκαν δύο εντελώς αχρησιμοποίητα, ξεπερασμένα αντίγραφα αρχείων (`settings.js` και `db_manager.py` στη ρίζα του project) που είχαν ξεμείνει από παλιότερη δομή — επιβεβαιώθηκε ότι η εφαρμογή φορτώνει αποκλειστικά τα πραγματικά αρχεία (`src/pages/settings/settings.js`, `database/db_manager.py`).

Όλα επαληθεύτηκαν end-to-end σε live Electron app πριν την ενσωμάτωση (προσομοίωση κατεστραμμένου backup, αποτυχία cloud sync, crash σε Archive Mode, κατεστραμμένη ημερομηνία CE, μη έγκυρες ημερομηνίες, ταυτόχρονες κλήσεις PDF).

## v1.1.26 — 2026-07-04

**Cleanup: εσωτερικές διορθώσεις από code review (Gemini + επαλήθευση)**

Σειρά μικρών εσωτερικών διορθώσεων μετά από code review, όλες επαληθευμένες ζωντανά πριν την ενσωμάτωση:

- **Fix: `ReferenceError` στο Puppeteer fallback του `generate-report-pdf`** — η μεταβλητή `tmp` δεν οριζόταν πουθενά (μόνο `os.tmpdir()` υπάρχει), οπότε αυτό το (σπάνια χρησιμοποιούμενο) fallback path θα έκανε πάντα crash αν ενεργοποιούνταν.
- **Fix: το `cloud-sync` αγνοούσε σφάλματα δικτύου στο PDF sync** — ο έλεγχος `isNetworkError` γινόταν μόνο για το backup sync (r1), όχι για το PDF sync (r2)· αν το backup πετύχαινε αλλά το PDF sync απέτυχε λόγω δικτύου, η συνάρτηση επέστρεφε ψευδώς επιτυχία.
- **Fix: το `performCleanStart` έσβηνε ρυθμίσεις που δεν έπρεπε** — το reset του config κρατούσε μόνο το `cloudRemotePath`, σβήνοντας σιωπηλά τις ρυθμίσεις retention (`cloudRetentionDays`, `cloudRetentionAutoEnabled`) και το ιστορικό sync (`cloudLastSync`, `cloudLastSyncStatus`).
- **Fix: `webSecurity: false` → `true`** σε όλα τα παράθυρα της εφαρμογής — επαληθεύτηκε ότι η εφαρμογή δεν φορτώνει ποτέ εξωτερικό περιεχόμενο (μόνο τοπικά αρχεία), οπότε δεν υπήρχε λόγος να είναι απενεργοποιημένο.
- **Fix: `_pyPending` timers δεν καθαρίζονταν όταν η απάντηση ερχόταν νωρίς** — διορθώθηκε σε 5 σημεία με το ίδιο guard pattern που ήδη υπήρχε στο κύριο IPC handler.
- **Fix: race condition στην εκκίνηση** — οι αρχικές κλήσεις δεδομένων (προϊόντα, τεχνικοί) περίμεναν κανονικά το python-ready σήμα πριν εκτελεστούν, αντί να ξεκινούν παράλληλα και να κινδυνεύουν με timeout σε αργή εκκίνηση backend.
- Μικρότερες: `_fileHash` πλέον διαβάζει αρχεία με streaming αντί να τα φορτώνει ολόκληρα στη μνήμη· `pyProcess.kill()` ελέγχει πρώτα αν η διεργασία είναι ήδη τερματισμένη· αφαιρέθηκε απαρχαιωμένο/διπλότυπο ιστορικό εκδόσεων από τα σχόλια στην κορυφή του main.js.

Όλες οι αλλαγές επαληθεύτηκαν end-to-end σε live Electron app (φυσιολογική εκκίνηση, αργή/timeout εκκίνηση, cloud sync, backup dedup, Clean Start).

## v1.1.25 — 2026-07-04

**Νέο: Διατήρηση Backup στο Cloud (configurable, με ασφάλεια πολλαπλών εγκαταστάσεων)**

- Νέα ρύθμιση στο Cloud Sync: αριθμός ημερών διατήρησης backup στο cloud remote (προεπιλογή 90). Λύνει την παλιά εκκρεμότητα ότι τα backups στο remote δεν κλαδεύονταν ποτέ μετά το sync→copy fix του v1.1.9.
- Επειδή διαχειριστής και χειριστής μοιράζονται το ίδιο cloud remote, ο αυτόματος καθαρισμός προστατεύεται από ένα **lock file** (`.retention-lock.json`) πάνω στο ίδιο το remote — μόνο ΜΙΑ εγκατάσταση μπορεί να έχει ενεργό τον αυτόματο καθαρισμό τη φορά. Αν μια δεύτερη συσκευή προσπαθήσει να τον ενεργοποιήσει, βλέπει ποια συσκευή τον έχει ήδη και πότε.

- Escape hatch "Εξαναγκασμένη Απελευθέρωση" για την περίπτωση που η συσκευή-κάτοχος του lock δεν υπάρχει πια.
- Επαληθεύτηκε πλήρες end-to-end σε live Electron app: κανονικός κύκλος ενεργοποίησης/απενεργοποίησης, άρνηση όταν το lock ανήκει αλλού (προσομοιώθηκε δεύτερη συσκευή), εξαναγκασμένη απελευθέρωση, προεπισκόπηση.

## v1.1.20 — 2026-07-03

**Fix: το update-checker δεν έδειχνε ποτέ νέα έκδοση («δεν βγάζει την αναβάθμιση στην εκκίνηση»)**
- Το `checkForUpdates()` (main.js) χρησιμοποιεί το Electron `net.request()` (Chromium network stack) για να ρωτήσει το GitHub API `/releases/latest`. Το GitHub στέλνει `Cache-Control: max-age=60`, αλλά το Chromium disk HTTP cache του Electron **δεν το επανα-επικύρωνε σωστά μεταξύ επανεκκινήσεων της εφαρμογής** — η πρώτη ποτέ απάντηση έμενε αποθηκευμένη επ' άπειρον, πολύ πέρα από τα δηλωμένα 60 δευτερόλεπτα.
- Αναπαράχθηκε τοπικά: μετά τη δημοσίευση της v1.1.19, η εφαρμογή (τρέχοντας ως v1.1.18) συνέχιζε να λαμβάνει `latestTag=1.1.18` σε δύο ξεχωριστές εκκινήσεις, λεπτά μετά τη δημοσίευση — ένα καθαρό `curl` την ίδια στιγμή έφερνε σωστά `v1.1.19`. Διαγραφή του φακέλου `Cache` της εφαρμογής επιβεβαίωσε αμέσως το πρόβλημα (μετά τη διαγραφή, η σωστή έκδοση εμφανίστηκε).
- Fix: το URL του request περιλαμβάνει πλέον cache-busting query param (`?_=${Date.now()}`) — μοναδικό URL σε κάθε κλήση, άρα το Chromium ποτέ δεν μπορεί να ταιριάξει με παλιά cached καταχώρηση. Επαληθεύτηκε: 2 διαδοχικές εκκινήσεις έφεραν σωστά `v1.1.19` ακόμα και με προϋπάρχον cache.
- **Σημαντικό follow-up:** αν ο χειριστής (ή οποιαδήποτε ήδη εγκατεστημένη έκδοση) έχει ήδη "κολλημένο" cache από πριν, το ίδιο το v1.1.20 installer ενδέχεται αρχικά να μην φανεί μέσω του built-in update-checker (ίδιο πρόβλημα, ένα τελευταίο κύκλο) — χρειάζεται χειροκίνητη εγκατάσταση για αυτή τη μία φορά.

## Εκκρεμότητες

### Μικρές / γνωστές

- **Επιλεκτική επαναφορά backup** — η τωρινή restore αντικαθιστά ολόκληρη τη βάση. Θα ήταν καλύτερα: επιλέγεις backup, βλέπεις τι δείγματα/δοκιμές έχει, φέρνεις μόνο συγκεκριμένα.
- **Διάδοση επεξεργασίας/διαγραφής στη βιβλιοθήκη εγγράφων** — το additions-only sync (v1.1.11) δεν διαδίδει edits/deletes ακόμα, χρειάζεται soft-delete tombstones.
- **Remote backup retention** — μετά το fix v1.1.9 (sync→copy), τα backups στο cloud remote δεν κλαδεύονται πια αυτόματα (μεγαλώνουν επ' άπειρον). Αποδεκτό προς το παρόν.
- **Επικοινωνία με χειριστή — εκκρεμεί feedback (2026-07-02)**: "Σφάλμα αποθήκευσης PDF" κατά την Αποθήκευση από Εκθέσεις στο μηχάνημα του χειριστή (στον διαχειριστή δουλεύει κανονικά) — πιθανή αιτία, ο τοπικός Φάκελος Δεδομένων στον χειριστή δείχνει σε μη-προσβάσιμο path (βλ. `checkDataFolderMismatch`, μόνο toast προειδοποίησης, όχι auto-fix). Διαγνωστικό fix v1.1.12 δείχνει πλέον το πραγματικό μήνυμα σφάλματος. Εκκρεμεί: screenshot από τον χειριστή του νέου μηνύματος + του Φακέλου Δεδομένων στις Ρυθμίσεις. Ο χειριστής ανέφερε επίσης ότι το fix ημερομηνίας υποπεριόδου (v1.1.10) "δεν λειτούργησε καλά" — εκκρεμούν λεπτομέρειες.
- **ESM Redesign** — CommonJS → ESM, αναβάθμιση Electron 28→latest, puppeteer 21→latest (το puppeteer 22+ είναι ESM-only, γι' αυτό είναι pinned στο v21 προς το παρόν).
